# Глобальные правила проекта

## Язык и стиль ответов

- Всегда отвечай **на русском языке**.
- Формулируй коротко, понятно и по делу.
- Если есть неоднозначность — задавай уточняющий вопрос.

## Правила кода

- Использовать только **TypeScript**.
- Не применять `any`, `unknown`, `ts-ignore` без прямого обоснования.
- Соблюдать правила **ESLint** и **Prettier**. Если после правок возникли ошибки — исправить.
- Не оставлять временный код: `console.log`, `debugger`, закомментированные фрагменты.
- Имена переменных, компонентов и функций — понятные и осмысленные.
- Сохранять существующий стиль и архитектуру проекта.
- Не менять структуру директорий без команды.

## Чистота и архитектура

- Компоненты — чистые и простые. Бизнес-логика — в хуках, сервисах, сторах.
- Избегай дублирования кода. Если логика повторяется — вынеси в функцию/хук/утилиту.
- Все публичные функции и сложные места — с понятным описанием «зачем» (краткий комментарий).

## Работа с изменениями

- Любые изменения показывай через **чистый diff**.
- Не изменяй файлы, не связанные с задачей.
- Не выполняй «оптимизацию ради оптимизации» без прямого запроса.

## Производительность

- Избегай тяжелых вычислений в рендере.
- Используй `useMemo`, `useCallback` или вынеси вычисления вне компонента.
- Если проект поддерживает — стремись к lazy-loading и code splitting.

## Безопасность и API

- Не выводи чувствительные данные в консоль или логи.
- Все запросы — через принятый в проекте API-клиент.
- Ошибки обрабатывай аккуратно: уведомление / безопасное состояние.

## Когда нужна ясность

- Если нет полной уверенности, какие файлы менять, — спроси.
- Если задача может быть выполнена разными способами — предложи варианты.

---

## Итог

Сохраняй аккуратность, простоту и предсказуемость кода. Поддерживай архитектуру проекта и используй существующие соглашения.
